# 电商测试数据字典

> 数据源: `demo`（嵌入式 SQLite）
> 数据范围: 2024-07-01 ~ 2028-06-30
> 生成方式: `src/data_generation.py`（Faker + 随机生成，seed=42）

---

## 表结构

### 1. categories — 商品分类

| 字段 | 类型 | 说明 |
|------|------|------|
| `category_id` | INTEGER PK | 分类 ID，1~10 |
| `name` | TEXT | 分类名称（如 Electronics类、Home类） |

- 固定 10 个分类

### 2. products — 商品

| 字段 | 类型 | 说明 |
|------|------|------|
| `product_id` | INTEGER PK | 商品 ID，1~200 |
| `category_id` | INTEGER FK → categories | 所属分类 |
| `name` | TEXT | 商品名称 |
| `price` | REAL | 商品单价（元），10% 高价值品 500-5000，90% 普品 10-500 |

- 共 200 个商品
- 价格呈**左偏分布**：少数高价值商品，多数中低价位

### 3. users — 用户

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | INTEGER PK | 用户 ID，1~5000 |
| `name` | TEXT | 用户姓名（中文） |
| `vip_level` | INTEGER | VIP 等级：0=普通(60%)，1=银卡(25%)，2=金卡(10%)，3=钻石(5%) |
| `register_date` | DATE | 注册日期，最早 2023-07-01 |

- 共 5000 个用户
- VIP 等级分布：60%/25%/10%/5%

### 4. orders — 订单

| 字段 | 类型 | 说明 |
|------|------|------|
| `order_id` | INTEGER PK | 订单 ID，1~50000 |
| `user_id` | INTEGER FK → users | 下单用户 |
| `order_date` | DATE | 下单日期 |
| `status` | TEXT | 状态：`completed`(85%)，`cancelled`(10%)，`refunded`(5%) |

- 共 50000 个订单
- 时间范围 2024-07-01 ~ 2028-06-30
- **销售额统计必须过滤 `status = 'completed'`**，排除 cancelled 和 refunded

### 5. order_items — 订单明细

| 字段 | 类型 | 说明 |
|------|------|------|
| `item_id` | INTEGER PK | 明细 ID |
| `order_id` | INTEGER FK → orders | 所属订单 |
| `product_id` | INTEGER FK → products | 商品 |
| `quantity` | INTEGER | 购买数量，1~5 |
| `unit_price` | REAL | 下单时单价（快照值，等于 products.price） |

- 每个订单 1~5 件商品，平均 3 件，约 **15 万行**
- `quantity * unit_price` 计算出单行金额

### 6. user_level_log — 用户等级变更记录

| 字段 | 类型 | 说明 |
|------|------|------|
| `log_id` | INTEGER PK | 记录 ID，1~2000 |
| `user_id` | INTEGER FK → users | 用户 |
| `old_level` | INTEGER | 变更前等级 |
| `new_level` | INTEGER | 变更后等级 |
| `change_date` | DATE | 变更日期 |

- 共 2000 条记录
- `new_level > old_level`（只记录升级）
- 变更日期在用户注册之后

## 表关联关系

```
categories ──1:N──→ products ──1:N──→ order_items ←──N:1── orders ←──N:1── users
                                                                    ↑
                         user_level_log ←──N:1── users ────────────┘
```

**JOIN 路径**：
- 订单金额 = `SUM(order_items.quantity * order_items.unit_price)` JOIN `orders` ON `order_id`
- 商品销售 = `order_items` JOIN `products` ON `product_id` JOIN `categories` ON `category_id`
- 用户升级 = `users` JOIN `user_level_log` ON `user_id`

## 核心业务指标口径

| 指标 | 定义 | SQL 要点 |
|------|------|----------|
| 销售额 (GMV) | 已完成订单的金额合计 | `SUM(quantity * unit_price)` WHERE `status = 'completed'` |
| 客单价 | 销售额 / 有订单的用户数 | 分子同上，分母 `COUNT(DISTINCT user_id)` |
| 订单量 | 已完成订单数 | `COUNT(DISTINCT order_id)` WHERE `status = 'completed'` |
| 复购率 | 下单≥2次的用户占比 | `COUNT(DISTINCT user_id) FILTER(cnt>=2)` / `COUNT(DISTINCT user_id)` |
| 退款率 | 退款订单数 / 总订单数 | `COUNT(*) FILTER(status='refunded')` / `COUNT(*)` |
| ARPU | 销售额 / 总用户数 | 同上 |
| VIP 升级率 | 升级记录数 / 活跃用户数 | `COUNT(*)` from `user_level_log` / `COUNT(DISTINCT user_id)` from `users` |

## 使用注意事项

1. **销售额计算必须过滤状态**：`orders.status = 'completed'`，否则会包含退款和取消订单。
2. **列名精确大小写**：`user_id`、`order_id`、`product_id` 等均为小写。不存在 `created_at` 列，日期字段为 `order_date` 和 `register_date`。
3. **时间过滤用字符串比较**：日期字段存储为 `YYYY-MM-DD` 字符串，用 `>= '2026-06-01' AND < '2026-07-01'` 过滤月份。
4. **无外键约束**：表之间逻辑关联但无数据库级外键，JOIN 时注意数据一致性。
5. **金额精度**：`price` 和 `unit_price` 为 REAL 类型，大额汇总建议用 `ROUND(..., 2)`。
